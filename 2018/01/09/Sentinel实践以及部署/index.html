Redis实例集合的高可用Sentinel(三)实践以及部署

发表于 2018-01-09 | 分类于 编程 | 阅读次数:

架构分析

目标：

满足Redis单实例的高可用，这里只推荐一种HA解决方案

步骤：

1.首先完成Redis的主从配置
2.搭建Sentinel的集群，最好为奇数，最低三个。
3.做VIP漂移，当故障自动迁移时，切换虚拟IP到新的master

底层是 Redis Sentinel 集群，代理着 Redis 主从，Web 端通过 VIP 提供服务。在部署 Redis 主从的时候，需要将虚拟 IP 绑定到当前的 Redis 主节点。当主节点发生故障，比如机器故障、Redis 节点故障或者网络不可达，Sentinel 集群会调用 client-reconfig-script 配置的脚本，将 VIP 漂移到新的主节点上。

优点：

1. 秒级切换，在 5s 内完成整个切换操作
2. 脚本自定义，架构可控
3. 对应用透明，当Redis服务切换时，应用不行参与

缺点：

Sentinel 模式存在短时间的服务不可用

环境

这里我们假设一个机器多个Redis实例做高可用，可能比较常见。当然单个方法同样适用。

假设资源有限只有：

1. 二台服务器 172.16.1.11、172.16.1.12，
2.虚拟IP提供Redis统一对外服务172.16.1.13
3.Redis master 有二台均部署在172.16.1.11上，发别为6379，6380端口。
4.172.16.1.12做二个Slave分别为6379，6380.

主从备份

Redis的主从很简单使用Slaveof即可，具体参考记录主从的文字。

新建三个Sentinel配置

从上可以看到，我们分别在机器172.16.1.11部署了一台Sentinel，在12上部署了二台Sentinel。

172.16.1.11是Redis Master所在服务器。三台Sentinel均监听172.16.1.11上的二个Master Redis实例。

因为172.16.1.11一台机器上有二个Master实例，所以当Redis故障的时候，需要全部转移，因此failover脚本只需要一个。

当172.16.1.11故障时，二台Slave的Sentinel依然可以投票决定进行Master故障转移。
